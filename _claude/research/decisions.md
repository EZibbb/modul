## ✅ ЗАКРЫТЫЕ РЕШЕНИЯ

### D000 — Старт проекта из CLAUDE_TEMPLATE
**Дата:** 2026-06-01
**Контекст:** новый проект Modul comp стартует с готовой папки CLAUDE_TEMPLATE, содержащей выверенную структуру Claude Code setup (правила, hooks, скиллы, методология).
**Решение:** копирование шаблона как стартовая точка. Placeholders заменены, memory rules перенесены, hooks активны.
**Последствия:** время на настройку среды = минуты вместо дней. Все наработанные паттерны (LOG-дисциплина, аудит после порции, 6-шаговый отчёт, push каждые 5 коммитов, graphify integration) активны с первого дня.

### D001 — Цель проекта: рабочее демо-презентация, не production
**Дата:** 2026-06-01
**Контекст:** Евгений планирует устроиться/зайти контрактом в компанию, которая будет торговать телеком-оборудованием (в т.ч. перекупные позиции). Проект — это **презентация «вот что я готов вам дать»**, а не сразу боевой магазин. Реальной 1С/ERP/склада ещё нет.
**Варианты:**
- Гибрид — 2-3 killer-фичи живые, остальное статичные экраны (быстро, баланс)
- Кликабельный прототип — всё имитируется (макс. визуал, мин. время, риск «не живое»)
- Полное рабочее демо — все основные фичи работают на mock-БД (макс. эффект «готово», дольше)
**Решение:** Выбран **полное рабочее демо** на mock-БД.
**Обоснование:** цель — убедить ЛПР, что Евгений может дать реально работающий продукт под их домен. «Оно реально работает» сильнее любого статичного макета; на встрече могут «понажимать».
**Последствия:**
- Стек ориентируем на быстрый, но настоящий веб-продукт с БД и живым деплоем (ссылка для встречи). НЕ Битрикс, НЕ 1С-интеграция на этом этапе.
- Данные — seed mock-каталог (реалистичные SFP/QSFP позиции + матрица совместимости + вендоры).
- Production-темы из ТЗ (импорт из 1С, DDoS, оптовые цены per-client) → откладываем, упоминаем как «дорожную карту» в презентации.
- Влияет на D-002 (стек), D-003 (объём демо), D-004 (деплой).
**Память:** см. [[owner-evgeny]]

### D002 — Уровень качества: настоящий продукт на seed-данных (уточняет D001)
**Дата:** 2026-06-01
**Контекст:** времени достаточно; Евгений хочет «прям хорошее», не одноразовую пустышку. Возник вопрос «демо или боевой продукт».
**Ключевая мысль:** «демо vs продукт» — ложная развилка из двух осей: **качество (craft)** можно выкрутить на максимум уже сейчас; **боевитость (live)** — нельзя в одиночку, т.к. завязана на данные/оплату/юрлицо компании, куда Евгений ещё не вошёл.
**Варианты:** настоящий продукт на seed-данных / лёгкое одноразовое демо / сразу боевой магазин (недостижим соло).
**Решение:** **настоящий продукт production-уровня на seed-данных**, спроектированный под боевой запуск.
**Обоснование:** строго лучше обоих краёв — сильнейший питч («рабочий продукт + как он подключается к вашей 1С») и не выбрасывается (растёт в боевой после трудоустройства подключением реальных источников данных).
**Последствия:** архитектура и код — production-grade; seed-данные = нормальный этап, не компромисс; production-темы (1С-импорт, оплата, 152-ФЗ, оферта) → roadmap/слайд питча.

### D003 — Стек проекта
**Дата:** 2026-06-01
**Контекст:** уровень = production-продукт (D002), значит стек должен уметь расти в боевой, а не только «показать».
**Решение:** Next.js (App Router) + TypeScript · Tailwind + shadcn/ui · Prisma ORM (SQLite в деве → Postgres/Neon в проде) · Claude API (Anthropic SDK) + RAG по seed-каталогу · Zod · деплой Vercel + Neon.
**Обоснование:** один репозиторий фронт+API, мгновенный деплой ссылкой, Prisma даёт безболезненный переход SQLite→Postgres, Claude API закрывает весь ИИ-слой по-настоящему.
**Последствия:** закрывает D-002 (стек) и D-004 (деплой). Claude API требует ключ+интернет на встрече → заложить запасной офлайн-режим для ИИ (D-006).
