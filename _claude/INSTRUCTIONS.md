## 💾 БЭКАП — КОГДА ДЕЛАЕМ И КОГДА НЕТ

> **Принцип:** бэкап = снимок принципиального состояния, не повседневный лог.
> Дублирующие бэкапы засоряют папку и теряют сигнал «здесь было важное событие».

### ✅ ДЕЛАЕМ БЭКАП

- Создан / финализирован **принципиальный документ** (concept/overview, architecture, monetization, roadmap, ТЗ агентов)
- **Переименование** или перестройка папок проекта
- **Завершение блока** из active.md (закрыта подсистема или этап)
- Перед **опасным действием** (массовое удаление, перезапись большого числа файлов)
- Явный запрос от Евгения — команда `БЭКАП`

### ❌ НЕ ДЕЛАЕМ БЭКАП

- Обновление `@last` в SESSION_CONTEXT.md
- Добавление / перенос задачи в `tasks/active.md` или `tasks/done.md`
- Мелкие правки текста (слово, абзац, формат таблицы)
- Обычная рутина в течение сессии
- Если за сессию **уже был** бэкап важного действия — повторно не нужно

### ПРАВИЛО ОДНОГО БЭКАПА В СЕССИЮ

В одной сессии — **максимум один бэкап** на финал важного блока. Повторные не делаем.

### ИМЯ ПАПКИ БЭКАПА

Формат: `backups/decisions/YYYY-MM-DD_короткое_описание/`
В папке — обязательно `_АКТУАЛЬНЫЙ.md` с описанием **что именно** в этом бэкапе.

**Ключевое:**

- `_claude/tasks/done.md` — все выполненные задачи с описанием что сделали

- После выполнения — `[x]` в active.md и подробная запись в done.md

**Статусы:** `[ ]` не начата · `[/]` в работе · `[x]` готово · `[?]` заблокирована · `[-]` отменена

**Атомарность:** задача дольше 30 минут — разбиваем на подзадачи.

**Обязательный отчёт после блока (6 шагов):**
```
✅ Шаг 1 — Анализ (что перечитал, с чем сравнил, что нашёл)
✅ Шаг 2 — Ошибки (таблица или «не найдено»)
✅ Шаг 3 — Исправления
✅ Шаг 4 — Повторная проверка
✅ Шаг 5 — Полнота (всё ли из обсуждения отражено)
✅ Шаг 6 — Финал (затронуто файлов: N)
```

**Жёсткие запреты:**
- ЗАПРЕЩЕНО писать «Готово» без 6 шагов
- ЗАПРЕЩЕНО объединять шаги
- ЗАПРЕЩЕНО начинать задачу без чеклиста
- ЗАПРЕЩЕНО выходить за рабочую папку без разрешения

## АУДИТ ПОСЛЕ EDIT/WRITE — обязательно

После каждого Edit/Write — быстрый аудит изменённого участка ДО следующего шага. Не откладывать.

**Что искать:**
- Дубли селекторов / функций / разделов с конфликтующими свойствами
- Override-правила, перебивающие общее (особенно `padding: 0`, `margin: 0`)
- `!important` без явной причины (ставил для отладки — попробовать убрать)
- Закомментированный код-мусор (`/* old: ... */`, `// removed`)
- Inline-стили в JS-шаблонах, оставшиеся от экспериментов
- `console.log` / debug-выводы
- Опечатки и обрезанные фразы после правки

**Алгоритм:** после Edit → Grep по селектору/функции/разделу → если есть дубли или override → разобрать. Не верить «наверное чисто» — проверить.

**Особенно важно** в итеративных задачах (3+ попыток): каждая отвергнутая попытка оставляет след в коде/тексте.

## НАВИГАЦИЯ ПО КОДУ И ФАЙЛАМ — экономия токенов

### Маркерная система
В каждом большом файле ставим метки:
```
@section: OVERVIEW
@section: ARCHITECTURE
@section: END
```
Алгоритм:
1. `Grep "@section: НУЖНЫЙ_БЛОК"` → получаю номер строки
2. `Read(file, offset=N, limit=80)` — читаю только нужный кусок
3. НЕ читаю весь файл, если знаю что ищу

Список маркеров проекта — в `_claude/NAV_INDEX.md`.

### Восстановление контекста
- `Grep "@last" SESSION_CONTEXT.md` → строка → читаю только последнюю запись
- Большие файлы: сначала Grep по маркеру, потом Read с offset/limit
