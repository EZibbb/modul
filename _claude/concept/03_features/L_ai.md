# L. ИИ-слой (инженерный со-пилот) — спецификация (11 полей)

> 🔥 Главный дифференциатор от FS.com. Опирается на K (каталог), B (совместимость), G (база знаний). Требует Claude API → запасной офлайн-режим (D-006).

## 1. ID + название
**L. ИИ-слой** — инженерный со-пилот. Под-фичи: L1 тех-консультант (чат) · L2 подбор решения по задаче · L3 парсер конфига/ТЗ · L4 диагностика DOM · L5 генератор КП.

## 2. Краткое описание
ИИ, который не «болтает», а делает инженерную работу: консультирует по каталогу, собирает решение под задачу, читает конфиг коммутатора и предлагает модули, диагностирует DOM-логи.

## 3. Цель
Превратить сайт из витрины в инструмент. Показать работодателю, что Евгений умеет строить современный AI-продукт под их домен. Снять у клиента порог «я не уверен, что подойдёт».

## 4. Triggers / точки входа
- Плавающая кнопка «ИИ-консультант» на всех страницах.
- CTA из пустой выдачи каталога / пустого результата совместимости.
- Отдельные страницы инструментов: «Подбор решения», «Проверить конфиг», «Диагностика DOM».
- Контекстный вызов из карточки («спросить про этот модуль»).

## 5. Описание процесса
**Архитектура — Claude API + tool use (grounded retrieval), НЕ голый чат.** Claude не выдумывает характеристики, а вызывает инструменты к нашей БД и отвечает по реальным данным.

**Инструменты (function calling) к Prisma-БД:**
- `search_products(filters)` — фасетный поиск (мост к A/K)
- `get_product(sku)` — полная карточка
- `get_compatibility(vendor, model)` — совместимые модули по портам (мост к B)
- `calc_optical_budget(segments, transceiver)` — расчёт бюджета (мост к G)
- `decode_part_name(name)` / `lookup_itu_grid(...)` — справочники (M/G)

**L1 Тех-консультант:** вопрос → Claude вызывает нужные инструменты → ответ со ссылками на конкретные SKU. Стриминг ответа.
**L2 Подбор решения:** «соединить два ЦОД, 80 км, 100G» → Claude собирает комплект (модули + патч-корды + DWDM-мукс) + расчёт бюджета → кнопка «добавить комплект».
**L3 Парсер конфига/ТЗ:** вставка `show running-config`/`show inventory` или Excel-BOM → Claude извлекает порты/модели → `get_compatibility` → таблица «порт → рекомендованный модуль».
**L4 Диагностика DOM:** вставка вывода `show interface transceiver` → Claude парсит Rx/Tx power, ток лазера → диагноз («Rx −18 dBm ниже чувствительности → проверьте бюджет/коннектор»).
**L5 Генератор КП (P2):** корзина → PDF-КП с тех-обоснованием.

## 6. UI/UX
- Чат-панель (drawer справа), стриминг, «карточки-результаты» с товарами прямо в ответе (не просто текст).
- Инструменты (L2/L3/L4) — отдельные страницы с полем ввода + структурированным результатом + действиями (в корзину / в сравнение).
- Прозрачность: показываем, какие позиции из каталога предложены (ссылки на SKU), бейдж «подобрано ИИ».

## 7. Данные (модели / сущности / API)
- SDK: `@anthropic-ai/sdk`. Модель: **Sonnet** для чата (быстро/дёшево), **Opus** для сложного подбора/парсинга — роутинг по типу запроса.
- **Tool use** к Prisma-БД (инструменты выше). **Prompt caching** на system-промпт + краткий слепок каталога (экономия токенов). **Streaming** ответов.
- System-промпт: персона — инженер по оптическим линиям; правило «рекомендуй только из каталога, если нет — скажи; не выдумывай характеристики».
- `Conversation`(id, messages[]) — для истории чата (в ЛК — P1).
- База знаний (ITU-сетки, глоссарий, статьи) — текстовый контекст для общих вопросов.

## 8. Доступность
Публично (демо). Лимит запросов на сессию (анти-абьюз). В ЛК — история диалогов (P1).

## 9. Лимиты
- Rate-limit на IP/сессию (демо).
- Контекст: не грузить весь каталог в промпт — только через tool use + кэш-слепок.
- Размер вставляемого конфига/лога — ограничение по символам.

## 10. Этап реализации (тир)
- **P0:** L1 тех-консультант (RAG/tool use) · L2 подбор решения по задаче.
- **P1:** L3 парсер конфига/ТЗ · L4 диагностика DOM · история в ЛК.
- **P2:** L5 генератор КП.

## 11. Приоритет
**P0** (L1+L2) — герой демо. После K/B/A (нужны данные и инструменты).
