# B. Совместимость — спецификация (11 полей)

> Killer-фича + спина модели данных. От неё зависят каталог (A), карточка (C), ИИ-подбор (L).

## 1. ID + название
**B. Совместимость** — подбор модулей под оборудование заказчика «вендор → модель». Под-фичи: B1 экспресс-виджет · B2 страница подбора · B3 таблица в карточке · B4 модель данных.

## 2. Краткое описание
Инструмент, который по выбранному оборудованию заказчика (Cisco/Juniper/Huawei…) показывает, какие наши модули в него встанут и заработают — с разбивкой по портам и рекомендациями «основной/альтернатива».

## 3. Цель
Снять главную боль аудитории: «какой модуль совместим с моим коммутатором?». Это причина, по которой инженер вообще приходит на сайт. Конкуренты (FS.com) дают это частично — мы делаем глубже + с ИИ.

## 4. Triggers / точки входа
- Экспресс-виджет на главной и вверху каталога.
- Пункт меню «Совместимость» / «Подбор совместимости».
- Кнопка «Проверить совместимость с вашим оборудованием» в карточке товара.
- URL с параметром `?compatibleWith=<deviceModelId>` (шаринг, ИИ-подбор).

## 5. Описание процесса
**B1 — Экспресс-виджет:** 3 каскадных списка Вендор → Серия → Модель → переход в каталог с авто-фильтром `/catalog?compatibleWith=<id>`. Нет точного совпадения → все модули подходящих форм-факторов + плашка «Уточните совместимость».

**B2 — Страница подбора:** Вендор (логотипы + поиск) → тип устройства (коммутатор/маршрутизатор/медиаконвертер/серверная карта) → модель → таблица совместимых модулей, **сгруппированная по порт-группам** (Uplink 4×10G SFP+, Downlink 24×1G SFP), с ролью Основной/Альтернатива, ценой, наличием. Кнопка «Добавить комплект в корзину» собирает набор под всю модель.

**B3 — Таблица в карточке:** список протестированных вендоров/моделей с поиском внутри, мин. версией ПО, примечаниями. Ссылка открывает виджет, предзаполненный этим модулем.

## 6. UI/UX
- Каскадные селекты с поиском (combobox, shadcn). Логотипы вендоров.
- Результат — таблица, сгруппированная по порт-группам; бейдж `tested` (протестировано) vs `заявлено`.
- Пустой результат — не тупик: фолбэк на форм-фактор + подсказка + CTA к ИИ-консультанту.
- Мобайл: селекты в полный экран, таблица → карточки.

## 7. Данные (модели / сущности / API)
M2M «товар ↔ модель оборудования» с атрибутами на связи:
- **Vendor**(id, name, slug, logo)
- **DeviceSeries**(id, vendorId, name)
- **DeviceModel**(id, seriesId, name/PN, deviceType)
- **PortGroup**(id, deviceModelId, label, count, formFactor, speed)
- **Product**(sku, name, formFactor, speed, … — атрибуты в модуле K)
- **Compatibility**(productId, deviceModelId, role[Основной/Альтернатива], note, minSoftwareVersion, portGroupId?, tested[bool])

API: `GET /api/compat/vendors`, `/series?vendorId`, `/models?seriesId`, `GET /api/compat/result?deviceModelId` (группировка по портам), `GET /api/catalog?compatibleWith=<id>`.
Индексы: (productId), (deviceModelId), уникальность (productId, deviceModelId).

⚠️ **Совместимость ≠ прошивка.** Совместимость = «встанет и заработает». Прошивка (кодировка под вендора) = услуга на складе, отражается в `note` («требует прошивки под Cisco»). См. D-007.

## 8. Доступность
Публичная фича (без авторизации). Для авторизованных B2B в результатах — индивидуальные/оптовые цены вместо базовых.

## 9. Лимиты
- Один модуль → сотни DeviceModel → поиск + пагинация внутри таблицы совместимости в карточке.
- Seed для демо: ~6 вендоров × ~3–5 серий × ~3–5 моделей, связано с ~50–100 модулями.

## 10. Этап реализации (тир)
- **P0:** B4 модель данных · B1 виджет · B3 таблица в карточке · базовая B2-страница.
- **P1:** группировка по порт-группам в B2 · кнопка «Добавить комплект» · бейджи tested.

## 11. Приоритет
**P0** (внутри P0 — наивысший: фундамент данных делаем раньше всех остальных модулей, кроме K seed).
